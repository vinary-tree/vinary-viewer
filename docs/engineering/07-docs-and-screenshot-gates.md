# Documentation and screenshot gates

The documentation suite is a build artifact in its own right: its screenshots are
regenerated by a headless harness, its diagrams are rendered from PlantUML source
under render-regression greps, and its diagram catalog is kept honest by two shell
loops. This page documents `scripts/screenshots.cjs`, the PlantUML render gates,
the catalog-honesty loops, and the pinned diagram toolchain — including a live
version-drift caveat on this workstation.

> **Audience.** Read this when regenerating screenshots, adding or editing a
> diagram, or reviewing a documentation change that touches figures.

---

## 1. Screenshots — `scripts/screenshots.cjs`

[`scripts/screenshots.cjs`](../../scripts/screenshots.cjs) is an automated,
**headless** screenshot harness. It boots the real renderer
(`resources/public/index.html`) in an **800×600** Electron window, mocks the
main-process IPC seam (the same way [`test/electron-smoke.js`](../../test/electron-smoke.js)
does), drives each feature into view, and writes a deterministic **800×600 PNG per
scene** into [`docs/screenshots/`](../screenshots/).

Two design choices make the output authentic and reproducible:

- **Real content service for parsed kinds.** The office, table, log, and archive
  previews are produced by the **real** `src/vinary/main/content_service.js`, so
  those screenshots show genuine parsing output. The trivial kinds
  (markdown/source/image/pdf/mermaid/directory) are fed directly with the same
  payload shapes `main/service.cljs` sends.
- **Deterministic 1:1 pixel mapping.** The harness sets
  `force-device-scale-factor 1` (1 CSS px == 1 device px) so every capture is
  exactly 800×600, and disables hardware acceleration for headless stability. One
  scene — the in-app web view — is a native `WebContentsView` overlay, so the
  harness captures its own `webContents` and composites it onto the chrome with
  ImageMagick.

Run it three ways:

```bash
npm run screenshots            # assets:sync + pdfjs:sync + compile renderer, then xvfb-run … electron scripts/screenshots.cjs
npm run screenshots:display    # same, but on your real DISPLAY (no xvfb)
electron --no-sandbox scripts/screenshots.cjs --only=native-pdf,web-preview   # a subset, for iteration
```

`npm run screenshots` wraps the harness in `xvfb-run -a -s "-screen 0
1280x1024x24"`, so it regenerates the whole gallery on a **headless** machine (CI,
a server) with no physical display. The `--only=<scene,scene>` flag renders a
subset for fast iteration on one figure.

## 2. PlantUML render-regression gates

Every diagram in [`docs/diagrams/`](../diagrams/README.md) is authored as PlantUML
`.puml` source and rendered to `.svg`. The render is **not** complete when
`plantuml` exits 0 — because PlantUML lies about success in one specific way.

> **Why the gate exists.** PlantUML does **not** exit non-zero when it merely
> *deprecates* your syntax. It renders a yellow warning banner **into the SVG** and
> silently drops the styling it could not apply. Three activity diagrams once
> shipped in exactly that state — banners committed, all step colours lost —
> because nobody looked at the output.

So the render pipeline greps the *output* for the two failure signatures, and both
must come back empty:

```bash
plantuml -tsvg docs/diagrams/*.puml
grep -lF deprecated        docs/diagrams/*.svg   # must print nothing
grep -lF "contains errors" docs/diagrams/*.svg   # must print nothing
```

`deprecated` catches the silent-banner class (a deprecated form that still
renders); `contains errors` catches PlantUML's hard-error banner (a genuine syntax
error it drew into the image instead of failing on). A green `plantuml` exit plus
two empty greps is the actual "render succeeded" condition. This is a documentation
analogue of the code-side gates: like the release-only Electron crash
([03-test-strategy.md §5](03-test-strategy.md#5-the-dev-vs-release-smoke-split)),
the failure is invisible unless you inspect the artifact you shipped, not just the
exit code.

Any diagram this engineering pillar embeds — `container-five-builds`,
`activity-asset-vendoring`, `component-test-taxonomy`, `component-terminal-renderer`
— is authored centrally as a `.puml`, must be registered in the catalog (§3), and
must pass these two greps before its SVG is committed.

## 3. Diagram-catalog honesty loops

The diagram catalog in [`docs/diagrams/README.md §5`](../diagrams/README.md) is "a
claim about the filesystem", so it is checked against the filesystem by two shell
loops. Both must print nothing (modulo the handful of intentionally-superseded
design diagrams that no document embeds):

```bash
# 1. Every .puml appears in the catalog table.
for f in docs/diagrams/*.puml; do
  grep -qF "\`$(basename "$f")\`" docs/diagrams/README.md || echo "MISSING from catalog: $(basename "$f")"
done

# 2. Every .puml is embedded by at least one document.
for f in docs/diagrams/*.puml; do
  b=$(basename "$f" .puml)
  grep -rq --include='*.md' --exclude-dir=diagrams "$b.svg" docs/ || echo "NOT EMBEDDED: $b"
done
```

Loop 1 guards against a diagram that exists but is undocumented; loop 2 guards
against a diagram that is authored but orphaned (embedded by no page). Together
they keep the catalog a *complete and accurate* index rather than a stale wishlist.
When you add a diagram, add its row to the catalog table **and** embed its SVG in
the page that motivates it, or one of the two loops will flag it.

The embedding convention every page follows (this pillar included) is a linked SVG
plus a cited source path:

```markdown
![alt text](../diagrams/NAME.svg)

*Diagram source: [`../diagrams/NAME.puml`](../diagrams/NAME.puml).*
```

The `.puml` is the single source of truth; never hand-edit an `.svg`.

## 4. The pinned toolchain — and a live drift caveat

The committed SVGs are byte-reproducible **only** with a pinned renderer pair. The
catalog records the pin:

> **Pinned toolchain.** The committed SVGs are produced by **PlantUML `1.2026.5`**
> with **Graphviz `14.1.5`**. Re-rendering with this pair reproduces every
> committed SVG byte-for-byte, so `git status` stays clean unless a `.puml`
> actually changed.

There are currently **63** `.puml` sources and 63 rendered `.svg` files in
`docs/diagrams/` (the "50" figure in the catalog prose predates recent additions;
the reproducibility property is unchanged).

Two practical cautions, both verifiable on this workstation:

1. **Graphviz drift.** This workstation currently has **Graphviz `15.1.0`**
   (`dot -V`), not the pinned `14.1.5`. Graphviz lays out the `dot`-based diagram
   kinds — component, class, and state — so **re-rendering those kinds here can
   reflow the layout** and produce a non-byte-identical SVG even when the `.puml`
   is unchanged. Sequence, activity, and deployment diagrams do not use Graphviz
   and are unaffected. If you must re-render a component/class/state diagram,
   render with the pinned Graphviz (or review the SVG diff as a layout reflow, not
   a content change).
2. **PlantUML self-reports no version here.** The installed `plantuml -version`
   prints an unsubstituted `$version$` placeholder (an Arch packaging quirk), so
   the binary cannot confirm it is `1.2026.5`. This is precisely why the
   **byte-for-byte SVG reproducibility check is itself the version check**: if a
   re-render of an unchanged, non-`dot` `.puml` yields a byte-different SVG, the
   renderer is not the pinned one.

## 5. Where documentation gates fit

None of these gates is wired into an npm script or CI today (there is no CI config
in the repository — see [08-ci-and-validation-discipline.md](08-ci-and-validation-discipline.md)).
They are **manual, reviewer-run** checks, and the proposed CI matrix in
[08](08-ci-and-validation-discipline.md#3-a-concrete-ci-matrix-that-should-run)
recommends adding them as a `diagrams` job: render all `.puml`, run the two
render-regression greps, and run the two catalog-honesty loops. The screenshot
harness is a heavier, display-dependent job better run on demand than on every
push. Documenting the gates here — so a contributor can run them by hand before
opening a pull request — is the interim discipline.
