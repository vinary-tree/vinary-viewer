# Terminal build and launch

vinary-viewer ships **three** runnable modes from one launcher: the desktop GUI
(the default), a headless terminal renderer (`vv --cli`), and an interactive
terminal viewer (`vv --tui`). This page documents the `vv` launcher's mode
dispatch, how [`install.sh`](../../install.sh) builds all three modes, the
`dist/cli` and `dist/tui` artifacts, and the `compile:cli` / `compile:tui` (and
`release:*`) build scripts.

> **Audience.** Read this when changing the launcher, adding a terminal mode flag,
> or diagnosing why `vv --cli` cannot find a grammar or the graphics WASM. For the
> *theory* of the terminal as a second renderer, see
> [theory/10](../theory/10-terminal-rendering-second-renderer.md); for the decision,
> [ADR-0019](../design-decisions/0019-terminal-preview-layer.md).

---

## 1. The `vv` launcher — one command, five modes

`install.sh` writes a small Bash launcher to `${BIN}/vinary-viewer` and symlinks
`${BIN}/vv → vinary-viewer`. There is **no npm `bin` field**; the launcher is
generated by the installer with the repository path (`$REPO`) and version baked
in, and it *mode-dispatches* on the first argument. The generated `case`:

```bash
case "${1:-}" in
  --cli)        shift; exec node "$REPO/dist/cli/vv-cli.js" "$@" ;;
  --tui)        shift; exec node "$REPO/dist/tui/vv-tui.js" "$@" ;;
  --gui)        shift; exec "$REPO/node_modules/.bin/electron" "$REPO" "$@" ;;
  -h|--help)    cat <<'USAGE' … USAGE
                exit 0 ;;
  -V|--version) echo "vinary-viewer $VERSION" ;;
  *)            exec "$REPO/node_modules/.bin/electron" "$REPO" "$@" ;;
esac
```

| Invocation | Dispatches to | Notes |
|------------|---------------|-------|
| `vv [files…]` | Electron GUI (`node_modules/.bin/electron "$REPO" "$@"`) | The **default** case (`*`). All args pass through, one tab per file/URL. |
| `vv --gui [files…]` | Electron GUI | Explicit GUI; `--gui` is shifted, an accepted no-op that documents intent. |
| `vv --cli <file>` | `node dist/cli/vv-cli.js` | The headless terminal renderer. Pipe-friendly: `vv --cli x.md \| less`. |
| `vv --tui <file>` | `node dist/tui/vv-tui.js` | The interactive terminal pager (scroll · `/` find · `t` contents · `q` quit). |
| `vv --help` / `vv -h` | Prints usage, `exit 0` | Never launches a window. |
| `vv --version` / `vv -V` | Prints `vinary-viewer <VERSION>` | `$VERSION` is read from `package.json` at install time. |

The design keeps a single memorable entry point: the common case (`vv README.md`)
is the shortest, the terminal modes are one flag away, and `--help` / `--version`
answer without booting Electron. Per-mode help is reachable with `vv --cli --help`
and `vv --tui --help`.

## 2. The two terminal artifacts

The terminal modes are two `shadow-cljs` `:node-script` builds (see
[01-build-system.md](01-build-system.md)):

| Build | Entry | Output | Mode |
|-------|-------|--------|------|
| `cli` | `vinary.cli.core/main` | `dist/cli/vv-cli.js` | `vv --cli` — render once to stdout. |
| `tui` | `vinary.tui.core/main` | `dist/tui/vv-tui.js` | `vv --tui` — interactive full-screen pager. |

Both are **Electron-free**. They reuse the same IR front-ends, the same WPDA /
streaming core, and the same `content_service.js` file reader as the GUI, but
lower to ANSI through `vinary.ir.backend.ansi` instead of HTML — this is the
"second renderer over the shared spine" of
[theory/10](../theory/10-terminal-rendering-second-renderer.md). The terminal-
specific namespaces are small and focused:

| Namespace | Role |
|-----------|------|
| `vinary.cli.core` / `vinary.cli.render` | The `vv-cli` entry point and one-shot render-to-stdout. |
| `vinary.tui.core` | The `vv-tui` entry point and interactive loop. |
| `vinary.tui.{keys,state,viewport,find,toc,term}` | The pure keys→state→frame TUI pipeline plus terminal teardown. |
| `vinary.terminal.caps` | Terminal-capability detection (isatty, `NO_COLOR`, `TERM`, graphics support). |
| `vinary.terminal.graphics` | kitty/sixel image encoding; loads `resvg.wasm` from disk to rasterize SVGs. |
| `vinary.terminal.syntax` | tree-sitter syntax highlighting; loads `tree-sitter.wasm` + the vendored grammars from disk. |
| `vinary.terminal.pdf` | Headless pdf.js text-run reflow to ANSI prose. |
| `vinary.terminal.stream` | The terminal side of the streaming pull-loop (bounded-memory log streaming). |

Because `graphics` and `syntax` load their WASM **from disk relative to the
compiled script's `__dirname`**, the two terminal builds depend on the vendored
`resources/public/js/resvg.wasm` and `resources/public/js/tree-sitter.wasm` plus
the `resources/public/grammars/` tree — which is exactly why the terminal build
scripts sync grammars and graphics WASM (§3).

## 3. Build scripts — `compile:cli` / `compile:tui` and `release:*`

The terminal build scripts each prepend the two syncs the terminal tools need,
then compile (or release) the target:

```jsonc
// package.json (scripts)
"compile:cli": "npm run grammars:sync && npm run graphics:sync && shadow-cljs compile cli",
"release:cli": "npm run grammars:sync && npm run graphics:sync && shadow-cljs release cli",
"compile:tui": "npm run grammars:sync && npm run graphics:sync && shadow-cljs compile tui",
"release:tui": "npm run grammars:sync && npm run graphics:sync && shadow-cljs release tui",
"graphics:sync": "node scripts/sync-graphics-wasm.mjs",
```

`grammars:sync` builds the tree-sitter grammar WASMs + catalog and `graphics:sync`
copies `resvg.wasm` (see [02-asset-vendoring.md](02-asset-vendoring.md)). Like all
release builds, `release:cli` / `release:tui` use `:optimizations :simple` — the
ANSI path shares the unified/rehype/content-service interop that `:advanced`
property-renaming would break ([ADR-0016](../design-decisions/0016-main-process-simple-optimization.md)).

The terminal builds are exercised by the `test:cli` and `test:tui` scripts, which
compile the artifact and then run its Node smoke (see
[03-test-strategy.md §3.2](03-test-strategy.md#32-terminal-smokes-node-no-display)):

```jsonc
"test:cli": "npm run compile:cli && node test/cli-smoke.js && node test/graphics-smoke.js",
"test:tui": "npm run compile:tui && node test/tui-smoke.js",
```

## 4. How `install.sh` builds all three modes

The installer builds the GUI and both terminal modes in one run, syncing the
shared grammar/graphics assets **once** up front so they are not re-fetched per
terminal build. The relevant phases of [`install.sh`](../../install.sh):

```bash
# 1. dependencies
npm install --no-fund --no-audit

# 2. shared terminal-renderer assets, ONCE (grammars + graphics wasm)
#    --skip-existing makes warm re-installs fast + idempotent
npm run grammars:sync -- --skip-existing && npm run graphics:sync

# 3. GUI build (main + renderer)          — $VV_BUILD is `release` (default) or `compile`
npx shadow-cljs "$VV_BUILD" main renderer

# 4. terminal build (cli + tui), one shadow-cljs run, no redundant re-sync
npx shadow-cljs "$VV_BUILD" cli tui

# 5. write the launcher + symlink vv → vinary-viewer into $BIN
```

Two build-orchestration facts are worth internalizing:

1. **The assets are synced once, then both `shadow-cljs` runs skip re-syncing.**
   Steps 3 and 4 invoke `shadow-cljs` **directly** (`npx shadow-cljs … main
   renderer`, `npx shadow-cljs … cli tui`) rather than through the npm
   `compile`/`compile:cli`/`compile:tui` scripts, precisely so the grammar and
   graphics syncs from step 2 are not repeated. Running the npm scripts instead
   would re-prepend `grammars:sync` + `graphics:sync` to *each* of `cli` and `tui`
   — the very double-fetch the installer comment calls out.
2. **The GUI's source preview shares the same vendored grammars.** The GUI build
   (`main renderer`) does not sync grammars itself (its npm script syncs only
   `assets` + `pdfjs`), so the up-front step 2 is what makes tree-sitter source
   highlighting work in the GUI too. Grammars are a *shared* runtime asset, synced
   at the install level, not per build.

The full installer flow — env knobs like `BIN` and `VV_BUILD`, the launcher
generation, and the v0.1.0 migration notice — is documented in
[06-release-process.md](06-release-process.md).

## 5. Runtime capability degradation

The terminal modes degrade gracefully by *detecting* the terminal's capabilities
rather than assuming them (`vinary.terminal.caps`). Two contracts, both asserted
by the terminal smokes:

- **Colour.** When stdout is not a TTY (piped) or `NO_COLOR` is set, `vv --cli`
  emits **zero** SGR escape bytes — so `vv --cli x.md | less` is clean text
  ([`cli-smoke.js`](../../test/cli-smoke.js)).
- **Graphics.** kitty/sixel image escapes are emitted only when the terminal
  advertises support (or `--graphics kitty|sixel` forces it); otherwise an image
  degrades to a labelled placeholder with no escapes
  ([`graphics-smoke.js`](../../test/graphics-smoke.js)).

This capability-first design is why the terminal renderer is safe to pipe, safe to
redirect to a file, and safe on a terminal that supports neither colour nor
inline graphics.
